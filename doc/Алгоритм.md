# Алгоритм диагностики

Документ описывает полный цикл автоматической диагностики конфликтов модов — от запуска до финального отчёта.

## Общий цикл

1. Лаунчер открыт, пользователь запускает скрипт в терминале.
2. Перед кликом скрипт закрывает старые crash/Fabric окна, оставшиеся от прошлых запусков.
3. Скрипт ожидает, пока курсор будет наведён на кнопку запуска, и кликает по ней.
4. После клика скрипт ожидает один из исходов: вылет (окно краша), диалог Fabric (отсутствуют зависимости) или успешный запуск.
5. Детекция crash/Fabric окон приоритетно ограничена окнами процесса лаунчера (fallback — общий поиск), чтобы не ловить «чужие» окна системы.
6. Если вылета нет и в сессии остались изолированные моды — автоматически запускается Recovery (если этап включён).
7. Диалог с пользователем появляется только когда остаются нерешённые изоляты, Recovery отключён или исход запуска неоднозначен (Fabric/NoLaunch).
8. Для завершений с Fabric missing dependencies в конце сессии показывается выбор: `продолжить` / `откатить` / `завершить`.

## Этап 1: Базовый Анализ

Скрипт читает краш-лог и ищет имена модов в тексте ошибки.

- Если мод найден и существует в `game/mods` — изолирует его (перемещает в Legacy) и пробует запуск заново.
- Если мод найден, но его нет в `game/mods` — переходит к Mixin-анализу.
- Кандидаты обрабатываются в конфликтно-зависимом приоритете:
  - сначала по «весу конфликта» из Fabric/лога (количество и тип evidence-строк, связи с другими mod id),
  - затем по dependency-priority (уровень/tier и количество зависимых),
  - при равенстве — по дате изменения.

### Fabric conflict-priority (важно)

Для Fabric-диалогов применяется дополнительная эвристика, чтобы не удалять вторичные моды раньше root-cause.

- Если mod id фигурирует только как `Remove/Replace` (или `Fix: add/remove/replace`) и при этом явно «ссылается» на другой более конфликтный mod id, такой mod id помечается как secondary и **откладывается** в текущей итерации.
- Это снижает ложные удаления вида: Fabric предлагает удалить `iris`, но первичный конфликт создаёт `immersive_portals`, который всё равно будет изолирован первым.
- Отложенные secondary mod id (по приоритизации `Select-QuickIsolateJarsByTier`) пишутся в лог отдельной строкой (`Fabric conflict-priority deferred secondary mod IDs: ...`).

## Этап 2: Mixin-анализ

> Скрипт: `Analyze-MixinErrors.ps1`

Целевая стратегия: парсинг Mixin-ошибок из краш-лога. Дешёвая проверка — 1–2 запуска на ошибку. Запускается до наслоения.

### Парсинг ошибок

Скрипт ищет два типа строк в логе:

- **`Mixin apply for mod <mod_id> failed ... from mod ... -> <class>`** — мод попытался применить Mixin и провалился.
- **`@Mixin target <class> was not found ... from mod <mod_id>`** — целевой класс для Mixin не найден.

Ошибки `Mixin apply failed` обрабатываются первыми как более надёжный сигнал root-cause.

### Резолв mod ID → JAR

1. Сначала — через карту зависимостей (построенную из `fabric.mod.json`).
2. Если не резолвится — fallback-сканирование JAR-файлов:
   - Чтение `fabric.mod.json` у внешних и вложенных (`jars`) модов.
   - Построение lookup по `id`/`provides` и именам mixin-конфигов (`*.mixins.json`, `mixin.*.json`).

### Определение владельца target-класса

Для target-класса определяется мод-владелец эвристикой: сегменты полного имени класса сопоставляются с известными mod ID.

### Проверка

1. Убираем source-мод, запускаем клиент.
   - Стабильно → source-мод = виновник → в Legacy, возврат в основной цикл.
   - Крашится → восстанавливаем source, пробуем target-мод.
2. Убираем target-мод, запускаем клиент.
   - Стабильно → target-мод = виновник → в Legacy, возврат в основной цикл.
   - Крашится → восстанавливаем target, Mixin-анализ не помог → далее к Наслоению.

## Этап 3: Наслоение (Аддитивное)

> Скрипт: `Layer-Mods.ps1`

Аддитивная стратегия: начинаем с минимума (core-библиотеки), добавляем моды слоями.

### Классификация модов по зависимостям

Моды разбиваются на уровни по количеству зависящих от них модов:

| Уровень | Критерий | Роль |
|---------|----------|------|
| 4 (core) | от мода зависит >10 других | Core-библиотеки (Fabric API, Cloth Config и т.п.) |
| 3 | от мода зависит ≤10 | Популярные библиотеки |
| 2 | от мода зависит ≤3 | Моды с небольшим числом зависимых |
| 1 | от мода не зависит никто | Конечные моды |

Внутри каждого уровня моды сортируются от старых к новым по дате изменения.

### Проверка core-библиотек

Всё, кроме модов уровня 4, изолируется. Базовый запуск выполняется с core-библиотеками + hash-cached моды для ускорения.

- Если baseline падает — выполняется retry в strict core-only режиме.
- Если strict core-only тоже падает — стоп, проблема в core-библиотеках, требуется ручная диагностика.

### Основной процесс (уровень 3 → уровень 2 → уровень 1)

Моды добавляются экспоненциальными батчами: 1, 2, 4, 8, ...

**При диалоге Fabric** (отсутствуют зависимости):

- Скрипт ищет среди изолированных модов те, которые Fabric помечает как missing (строки `requires ... which is missing`), и восстанавливает их строго по mod-id.
- Если после восстановления зависимостей получаем Crash — запускается отдельная Crash-Изоляция внутри текущего батча (по requiring-кандидатам с учётом tier-приоритета).
- Если диалог не уходит после повторов — батч возвращается в карантин.
- После 3 подряд неразрешимых Fabric-батчей в одном уровне — уровень останавливается.
- Если были неразрешимые батчи — Наслоение завершается с кодом 3 (неполное), запускается fallback.

**При вылете:**

1. Базовый алгоритм: читаем лог, ищем виновный мод.
2. Если лог не помог — экспоненциальная бинарная Изоляция внутри батча (итеративная, поддерживает множественных виновников).
3. Для углублённой диагностики уровня 1: временно изолируем все уже активные уровень 1 моды, кроме проблемного батча.
4. После диагностики батча — возвращаем временно изолированные моды и делаем контрольный запуск.
5. Найденный виновник → в Legacy, Наслоение продолжается.

**При успешном запуске:**

- 20 секунд ожидания для подтверждения стабильности (60 с флагом `-LongLaunchTimeout`, алиас: `-ThoroughStabilityCheck`).
- Клиент закрывается, переход к следующему слою.
- Если игрок сам закрыл игру — батч считается чистым.
- На финальном батче последнего уровня — игра остаётся запущенной.

### Fallback

При полном провале Наслоения — переход к субтрактивной Изоляции.

## Этап 4: Изоляция (Гибридная субтрактивная)

> Скрипт: `Isolate-Incompatible-Mod.ps1`

Обратная стратегия (fallback): dependency-aware гибрид, который сочетает экспоненциальные пробы, бинарное сужение и линейную Изоляцию.

### Порядок и режимы

Моды разбиваются на dependency-aware группы (сначала меньший «вес» по зависимостям):

| Группа | Критерий |
|--------|----------|
| 1 | от мода не зависит никто (0) |
| 2 | от мода зависит ≤3 |
| 3 | от мода зависит ≤10 |
| 4 | core-библиотеки (остальные) |

Внутри группы — от новейших к старым по дате изменения.

Режим Изоляции по умолчанию:

- Для ранних групп используется экспоненциальный probe (`1, 2, 4, ...`) с бинарным уточнением до порога.
- Для поздних групп — линейная Изоляция (по одному моду).
- При Fabric dependency dialog скрипт:
  - восстанавливает удалённую missing-зависимость (если именно её удаление вызвало диалог),
  - quick-isolat'ит requiring-мод(ы),
  - продолжает цикл.
- Если подтверждён виновник уровня 2+ (по dependency-tier), в очередь исключения автоматически добавляются моды, которые зависят от него (транзитивно по карте зависимостей).

### Критерий остановки

Когда при очередном исключении происходит успешный запуск или подтверждённое изменение сигнатуры ошибки — фиксируется виновник/кандидат.

## Этап 5: Recovery

> Скрипт: `Recover-PhantomCulprits.ps1`

Пост-Изоляция: проверяет, не были ли найденные виновники «ложными» из-за общей Mixin-ошибки. Запускается после Наслоения/Изоляции при 3+ виновниках, а также после чистого запуска, если в текущей сессии ещё есть изолированные моды.

### Логика

1. Виновники группируются по совпадающей Mixin-ошибке (`CrashEvidenceKey`).
2. Для групп из 3+ виновников с одинаковой `@Mixin target` ошибкой:
   - Извлекается source/target мод из ошибки, резолвится в JAR.
   - Предполагаемый root-cause мод убирается из `game/mods`.
   - Все фантомные виновники группы восстанавливаются обратно.
   - Запуск клиента.
3. Если стабильно → root-cause подтверждён: переносим его в Legacy, виновники восстановлены.
4. Если крашится → откат: убираем виновников обратно, восстанавливаем root-cause.

## Финальный отчёт (Summary)

Отчёт включает:

- Время начала и окончания сессии, общее время работы.
- Виновники, разбитые по этапам (Mixin-анализ / Наслоение / Изоляция / Recovery).
- Восстановленные моды (Recovery) — отдельным списком со знаком `+`.
- Текущий список изолированных модов.

---

## Краткая схема (Summary)

```
Запуск → Краш?
  │
  ├─ Базовый Анализ → мод найден? → изолировать → запуск заново
  │
  ├─ Mixin-анализ → source/target мод → проверить по одному (1–2 запуска)
  │
  ├─ Наслоение → core → +уровень 3 → +уровень 2 → +уровень 1
  │   └─ При краше батча: лог-анализ → бинарный поиск внутри батча
  │
  ├─ Изоляция (fallback) → exp/binary + linear
  │
  └─ Recovery → группировка по Mixin-ошибке → проверка root-cause
```
